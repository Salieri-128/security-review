# EVM Call Stack

## 一句话理解

EVM 支持合约之间的同步调用栈。

一个函数执行到外部 `call` 时，当前函数不会立刻结束，而是先暂停，把控制权交给被调用的外部地址。等外部调用执行完并返回后，原函数才会继续往下执行。

## 正常调用流程

假设 A 合约调用 B 合约：

```solidity
contract A {
    function foo(address b) external {
        // 1. A 开始执行
        B(b).bar();
        // 3. B 返回后，A 继续执行
    }
}

contract B {
    function bar() external {
        // 2. B 执行
    }
}
```

调用流程可以理解为：

```text
A.foo()
  -> B.bar()
  <- B.bar() returns
A.foo() continues
```

也就是说，A 调用 B 时，A 的执行上下文还在，只是暂时停住了。

## 每次 CALL 都会创建新的执行上下文

在 EVM 中，外部调用会创建一个新的执行上下文。

新的执行上下文会有自己的：

- `msg.sender`
- `msg.value`
- calldata
- gas
- 当前执行的合约地址

旧的执行上下文会被保留在调用栈里，等待新的调用返回。

## 控制权转移

外部调用的本质是：当前合约主动把控制权交给另一个地址。

常见外部调用包括：

```solidity
target.call(data);
target.call{value: amount}("");
IERC20(token).transfer(to, amount);
otherContract.someFunction();
```

如果目标地址是合约，它就可以执行自己的代码。

如果目标合约在 `receive` 或 `fallback` 中写了逻辑，那么转 ETH 或调用不存在的函数时，也可能触发这些逻辑。

## 合约可以在调用栈中再调用回来

因为外部合约拿到了执行权，所以它可以再调用原来的合约。

调用路径可能变成：

```text
A.foo()
  -> B.receive()
      -> A.foo() or A.otherFunction()
```

这表示 A 的第一次执行还没有结束，B 又重新调用了 A。

这就是理解 reentrancy 的底层基础：不是 EVM 强制打断 A，而是 A 自己执行外部调用，把控制权交给了 B，B 再主动调用回来。

## 不是系统中断

这个过程不是操作系统那种“中断”。

EVM 不会无缘无故打断正在执行的函数。只有当合约代码主动执行外部调用时，控制权才会交出去。

更准确的理解是：

- 当前函数执行到外部 call
- 当前上下文暂停
- 被调用合约开始执行
- 被调用合约可以继续调用其他合约
- 最内层调用结束后，控制权一层层返回
- 原函数继续执行剩下的代码

## 和 receive / fallback 的关系

当合约收到 ETH 或收到无法匹配的函数调用时，可能触发 `receive` 或 `fallback`。

这两个函数也是普通代码入口，一旦被触发，就可以继续调用别的合约。

例如：

```text
A.withdraw()
  -> send ETH to B
      -> B.receive()
          -> B calls A again
```

所以，任何向外部地址发送 ETH 的代码，都要意识到：接收方如果是合约，它可能会执行代码，而不是单纯“收钱”。

## 审计时注意

看到外部调用时，要问自己：

- 当前函数在外部调用前，关键状态是否已经更新？
- 外部地址是否可能是合约？
- 目标合约的 `receive` 或 `fallback` 是否可能执行逻辑？
- 外部调用返回前，当前函数有哪些状态还处于“未完成”的中间状态？
- 外部合约能否在这个中间状态下调用回来？

## 我的理解

EVM 调用栈像一层层同步函数调用。

合约 A 调用合约 B 时，A 不是结束了，而是暂停等待 B 返回。B 拿到控制权后，可以继续调用其他合约，甚至调用回 A。

所以外部调用不是简单的“发消息”或“转账”，而是一次真正的控制权转移。理解这一点，是理解 reentrancy、fallback、receive 和外部调用风险的基础。
